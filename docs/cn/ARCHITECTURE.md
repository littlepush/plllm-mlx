## 核心组件

### 1. 模型加载器

**作用**: 处理模型生命周期和推理

**核心类**:
- `PlModelLoader` (抽象基类)
  - 定义所有加载器的接口
  - 通用功能（配置、流生成）
  - 对话补全方法
  
- `PlMlxModel` (MLX-LM)
  - 纯文本模型
  - 使用 `mlx_lm.stream_generate`
  - 标准的逐token生成
  
- `PlMlxVlmModel` (MLX-VLM)
  - 视觉语言模型
  - 使用 `mlx_vlm.generate.stream_generate`
  - 支持图像输入

### 2. Step 处理器

**作用**: 将原始生成结果转换为 API 响应

**处理流程**:
```
生成结果 → Step处理器 → PlChunk → SSE响应
```

**核心类**:
- `PlStepProcessor` (抽象基类)
  - token累积
  - 状态管理
  - 结束原因检测
  
- `PlBaseStepProcessor`
  - 基础文本处理
  - 工具调用检测
  
- `PlOpenAIStepProcessor`
  - OpenAI兼容格式化
  - 使用统计
  
- `PlQwen3ThinkingStepProcessor`
  - 思考模式支持
  - 推理内容处理

### 3. KV 缓存

**作用**: 通过前缀缓存实现高效的提示复用

**实现**:
```
消息链: [msg_id_1, msg_id_2, msg_id_3, ...]
                ↓
            缓存查找
                ↓
         部分匹配找到
                ↓
      复用前缀，预填充剩余部分
```

**核心类**:
- `PlMessageBasedKVCache`
  - 消息级缓存
  - 基于MD5的消息ID
  - LRU淘汰
  
- `PlChain`
  - 消息链表示
  - 缓存项引用
  
- `PlChainCache`
  - 基于OrderedDict的存储
  - 最长前缀匹配

### 4. 进程隔离

**作用**: 稳定的多模型服务

**架构**:
```
主进程 (API服务器)
    │
    ├─► 子进程1 (模型A)
    │   └─► 模型 + KV缓存
    │
    └─► 子进程2 (模型B)
        └─► 模型 + KV缓存
```

**通信**:
- 请求队列: 主进程 → 子进程
- 响应队列: 子进程 → 主进程
- 异步桥接: 队列 → AsyncGenerator

**优势**:
- 内存隔离
- 容错性
- 资源清理干净
- 并行服务

## 数据流

### 对话补全请求

```
1. 客户端请求
   ↓
2. 路由器 (chat.py)
   ↓
3. 本地模型管理器
   ↓
4. 进程管理器
   ↓
5. 子进程: 模型加载器
   ├─► prepare_prompt()
   │   └─► KV缓存查找
   ├─► stream_generate()
   │   ├─► MLX推理
   │   └─► Step处理器
   └─► 响应块
   ↓
6. SSE流
   ↓
7. 客户端响应
```

## 设计决策

### 为什么使用进程隔离？

**问题**: 同一进程中的多个模型
- 内存碎片
- 资源冲突
- 崩溃传播

**解决方案**: 独立的子进程
- 干净的内存管理
- 故障隔离
- 独立的生命周期

### 为什么使用基于消息的 KV 缓存？

**问题**: 提示缓存的挑战
- 粒度: token vs 消息？
- 匹配: 精确 vs 前缀？
- 效率: 存储 vs 速度？

**解决方案**: 消息级前缀缓存
- 自然的边界（消息）
- 前缀匹配用于多轮
- MD5用于快速比较
- LRU用于内存管理

### 为什么优先流式传输？

**问题**: 长生成时间
- 用户体验差
- 连接超时
- 资源浪费

**解决方案**: 实时流式传输
- 立即反馈
- 更好的用户体验
- 高效的资源使用

## 性能考虑

### 内存管理

**策略**:
1. 进程隔离
   - 每个模型: 独立进程
   - 干净关闭: 完全清理
   
2. KV缓存淘汰
   - LRU策略
   - 内存阈值
   - 最小条目数
   
3. 量化
   - 4-bit/8-bit模型
   - KV缓存量化
   - 减少内存占用
